## 0. Headline deviations (read first)

### 0.1 RQ-14 compose-pivot (inferops D-1, 2026-07-11 — carried through every IO-T00x task)

**This is the single most important thing to know before reading the rest of this document.**
The inferops build/execution environment cannot schedule *any* Kubernetes pod, on any
CRI-based distribution — proven at the runc/nsexec level
(`/home/user/tools/k8s-env-probe-report.md`, cited by `inferops/docs/implementation-notes.md`
Deviation D-1): containerd's CRI plugin unconditionally sets `oomScoreAdj: -998` on every pod
sandbox, and the sandbox's own container lacks `CAP_SYS_RESOURCE`, so
`nsexec: failed to update /proc/self/oom_score_adj: Permission denied` fails identically under
kind and k3s, under both runc and crun. This is an environment limitation, not a defect in any
manifest.

**User-approved (RQ-14, 2026-07-11) resolution, carried through every inferops task this
checklist cites:** the *runtime* operational stack (gateway, mock-backend, dev-PostgreSQL,
observability stack, llama.cpp engine) runs on **Docker Compose**, which this environment's
Docker daemon genuinely can execute containers for. Kubernetes manifests are still fully
authored (Kustomize + raw manifests, ADR-0001) and validated against a **live k3s API server**
(server-only, `--disable-agent`, no kubelet): real `kubectl apply` all the way to etcd, exercising
schema validation, defaulting, and controller reconciliation (Deployment → ReplicaSet → Pod,
StatefulSet → Pod, Service → ClusterIP) — everything except the final kubelet→CRI→runc
pod-start step, which this environment cannot execute for **any** Kubernetes distribution.

**Consequence for this checklist, stated plainly and never elided:**

- Every "running stack" / "deployed" / "serving traffic" claim below refers to the **Docker
  Compose stack**, not a scheduled Kubernetes Pod.
- Every "manifest correctness" claim below refers to **k3s API-server validation**
  (`kubectl apply` to etcd + server-side dry-run), not a live scheduling decision.
- **No pod scheduling is claimed anywhere in this document or in `pins/pins.yaml`.** Where the
  evidence below says a Deployment "reconciled correctly," that means ReplicaSet/Pod objects
  were created in etcd by the controller, not that a kubelet ever started a container from
  them.
- Per `inferops/docs/implementation-notes.md` D-1: this deviation does not change I5's
  acceptance criteria (deployment from released images, warm-up-aware readiness, zero-error
  rolling update, golden dashboards, end-to-end traces are all still produced — just on
  compose instead of a scheduled cluster) and was pre-approved by the user before any IO-T00x
  task began; it is not re-litigated here.

Evidence: `inferops/docs/implementation-notes.md` Deviations D-1 (2026-07-11) and its IO-T005
extension (2026-07-12); `inferops/clusters/local/evidence/k3s-validation-20260711.txt`;
`inferops/clusters/gpu-node/evidence/k3s-validation-20260712.txt`.

## 3. New evidence generated this session (light, consumer-side demonstration)

Per this task's brief ("OPTIONALLY run ONE clean end-to-end Scenario D demo"), this session ran
one light, real demonstration through the live stack rather than relying solely on citation —
specifically targeting the one verification gap the cited `inferops` evidence left open: the
observability pipeline (Prometheus metrics + Tempo trace resolution) had only been proven
end-to-end against the **mock-backed** main gateway (IO-T003); IO-T005's own `llamacpp-smoke.sh`
verified real inference + cancellation but did not check Prometheus/Tempo. This demo closes
that gap for the real llama.cpp path specifically, while staying light (7 total requests) since
the stack is shared with other work.

**What was run** (`evidence/i5/raw/demo-20260712T005146Z/`, this repo, this session):

1. Confirmed via `docker inspect` that the live `inferops-gateway-llamacpp` and `inferops-llama-cpp`
   containers run exactly the pinned digests (`sha256:1971426b393b3e00b30cac0690d38b31667b5e34ebbeb6e111a54c369fb54c7e`
   and `sha256:43af71918dda78a1daaf19849e1c3cccfd7bad7c432b6c1420a45a62e99410be` respectively) —
   `transcript.log` lines 1–10.
2. Baselined Prometheus's `inference_requests_total{backend="llamacpp"}` counters (46 successful
   requests already recorded from prior inferops sessions' own smoke/upgrade/rollback runs).
3. Sent 5 non-stream + 2 streaming real chat-completion requests to `http://127.0.0.1:8082/v1/chat/completions`
   (seed `5005001`/`5005002`, `qwen2.5-1.5b-instruct`): **7/7 HTTP 200**, both streams
   terminated with `data: [DONE]`, and the non-stream response content (`"acknowledged"`)
   matches the deterministic prompt exactly (`nonstream-1.json` .. `nonstream-5.json`,
   `stream-1.sse`, `stream-2.sse`) — genuine inference, not a stub.
4. Re-queried Prometheus: `inference_requests_total{backend="llamacpp",error_class="none"}`
   advanced **46 → 53** (delta 7, matching exactly the 7 requests sent) —
   `prom-requests-total-before.json` / `prom-requests-total-after.json`.
5. Queried Prometheus's exemplar API for `inference_ttft_seconds_bucket{backend="llamacpp"}`
   over the trailing 5 minutes: 2 exemplars found; took the newest, trace_id
   `1321511fc73589d8e766cd702c54c2a7` — `exemplars-ttft-llamacpp.json`.
6. Resolved that trace_id against Tempo (`docker exec inferops-gateway-llamacpp wget ... http://tempo:3200/api/traces/<id>`,
   the same method `inferops/scripts/verify-observability.sh` itself uses): the **full expected
   span sequence** was present — `recv, queue.wait, upstream.connect, ttft, stream.relay, settle`
   — and the `recv` span's `gen_ai.request.model` attribute correctly reads
   `qwen2.5-1.5b-instruct`, confirming the trace is genuinely from this real-engine request, not
   a stale/unrelated one — `tempo-trace-1321511fc73589d8e766cd702c54c2a7.json`.

**Result:** the observability pipeline (Prometheus scrape → exemplar → Tempo trace resolution,
full span sequence) is now verified end-to-end against **both** backends the stack runs (mock,
