## 2. Detection gap

This is not framed as an operator-missed-it incident — the mid-stream SSE error event itself
**is** the detection signal, delivered synchronously to the client within the same stream, at
the moment of failure, with no propagation delay. From an operator's perspective, the two
metrics that would surface this in a real incident both moved/held correctly:
`inference_requests_total{error_class=upstream_error}` incremented for every affected stream,
and `inference_retries_total` correctly stayed flat (see §3 — that flatness is a *feature*
here, not a missing signal). **No detection gap was found in this scenario** — recorded as a
clean-pass example, deliberately paired with PM-001 (which does have a real detection gap) to
show the difference between "the contract's failure-signaling design working as intended" and
"a mechanism that exists but doesn't yet fire."

## 3. Root cause

Not a defect — this postmortem documents a **deliberate, hard fault injection** (`docker kill
-SIGKILL`, no drain) to verify the gateway's contractually-required response. The structural
finding worth recording: infergate's STREAMING-state failure handler
(`internal/gateway/gateway.go:749-761`) is a code path entirely separate from the
DISPATCHED-state pre-first-token retry loop, and is never invoked once streaming has begun —
confirmed here observationally (0 retries across 5-6 post-first-token failures), not merely
by reading the source. `internal/gateway/reliability_test.go` documents this as a
"zero-post-first-token-retries proof"; this campaign's population run is independent,
black-box confirmation of that same guarantee under a real process kill.

## 4. Mitigation

`mock-faults` was restarted ~2 seconds after the kill (the test's own recovery step, not a
production mitigation). Throughout the kill/restart window, 51/60 requests in the same
concurrent population completed normally — no evidence that one killed backend held other
in-flight or newly-arriving streams hostage. Partial output for every interrupted stream was
billed correctly: the aggregate `+1796` output-token delta is consistent with each
interrupted stream's own `output_tokens` field in `events.jsonl` (2-3 tokens for the streams
caught mid-flight), i.e. **exactly the tokens actually delivered before the cut, no more, no
less**.

**What was NOT measured:** whether a real client SDK/library correctly surfaces this SSE
error event to an end user in a UX sense — out of gateway/campaign scope. Output *quality* is
not applicable here (no content is regenerated after a mid-stream failure — the contract
explicitly forbids re-generation on the client's behalf, and none was observed). The one
unexplained `upstream_timeout` (1/60, `output_tokens=120`) was investigated only to the level
of a plausible explanation (DNS-cache reuse of the same Docker network alias across the killed
and restarted `mock-faults` containers) — not conclusively root-caused, and explicitly not
folded into the "5 clean kill-attributable errors" count used for the verdict.
