## 2. Detection gap

Framed against "would an operator have known the database was down without this test?": the
request path itself gives **zero signal**, by design — settlement is intentionally asynchronous
and off the request path, so client-visible latency/errors cannot serve as a detection channel
for this class of failure (and correctly did not, in this run). The only channels that would
have surfaced the outage are infrastructure-level (container/process health, a
backlog-depth metric on the async settlement path, or the write layer's own error logs) —
**this campaign did not check whether such a metric or alert exists in the deployed stack**, a
coverage gap recorded honestly rather than assumed closed. There is no client-visible
detection gap, because — by the design this scenario verifies — there is nothing wrong from
the client's perspective during this fault. Any real detection would have to come from
watching the database or the settlement subsystem directly, not from request-path symptoms.

## 3. Root cause

Not a defect — this postmortem documents a **deliberate fault injection** (`docker stop`
against a real dependency, `inferops-postgres-dev`) to verify that the async-settlement design
decouples request serving from usage accounting, exactly as the gateway's own architecture
intends. No root cause to remediate; the "cause" of the outage is the injection itself.

## 4. Mitigation

`docker start inferops-postgres-dev` restored the database ~3 seconds after the stop. The
in-memory-buffered backlog — 15 usage records accumulated while the database was unreachable
— drained completely and idempotently within 1 second of recovery: the row-count delta
(+35) exactly accounts for every request sent across all three phases (0 lost), and the
idempotency spot-check found 0 duplicate settlements for any `request_id` (0 double-billed).

**What was NOT measured:** backlog behavior under an outage longer than ~3 seconds — this run
only exercised a short outage well inside whatever the backlog's configured bound is; whether
backpressure or record-dropping kicks in as the outage approaches that bound is untested here.
Also not measured: whether an operator-facing alert or metric exists for "usage-settlement
backlog depth" or "usage-DB connectivity" (§2) — this campaign checked the *database* state
directly via `psql`, not any gateway-exposed signal for the same condition.
